Demo 0

Work IQ — the boring magic everyone uses every day

Before the “multi-model, multi-agent, agentic skills” story, anchor on this: the daily reason 200 million people open Copilot Chat is not strategy — it’s the 30 seconds you save before every meeting, the email you find without scrolling, the reply you don’t have to write from scratch. Start here so the room sees themselves in the demo.

⏱️

How to run it

Open Copilot Chat. Don’t change a thing in the UI. Just paste the four prompts in order. Each one should take under 30 seconds to answer. The point is rhythm, not depth.

Catch me up

The morning ritual. Copilot reads inbox + Teams + meetings in one shot.

“This is the prompt I run every morning before my coffee is finished.” 
What did I miss yesterday afternoon and this morning? Anything urgent in my inbox or Teams I should look at before my next meeting.

Find that thing

Cross-app search the way it should always have worked. Files, chats, mails, all at once.

“I know someone shared that — but where? Copilot finds it across mail, chat and files in one go.” 
Someone shared something about EU heat pump policy in the last couple of weeks. I think it was in a Teams chat or maybe an email. Find it and give me the gist.

Draft the reply

Open any recent email in Outlook, click Copilot → Draft. Or do it from Chat.

“The 60 seconds it takes to write ‘thanks, will come back to you by Friday’ \u2014 gone.” 
Draft a short reply to the last email I got from my team about the AI programme. Thank them for the work, say I'll come back with feedback by Friday, keep it warm but not gushing. Three sentences max.

Recap the meeting I missed

Works against a recorded Teams meeting transcript. If no recent one exists, swap for “the meeting I had on Monday”.

“You don’t need to attend every meeting. You need the three things that came out of it.” 
I missed yesterday's leadership sync. What were the three things that actually got decided, and is there anything that needs me to follow up?

Talking points

  • This is the “why everyone uses it” layer — before any AI strategy conversation
  • Works against the data the user already has access to — nothing new to configure
  • Same data boundary, same Purview audit, same model access controls as the rest of Copilot
  • Quick wins build adoption; adoption is the prerequisite for everything that follows

Multi-Model Choice — one UI, many brains

Open Copilot Chat. Point at the model selector top-right. Explain in one sentence: “Copilot lets you pick which model thinks about the prompt — without changing the app, the data boundary, or the user experience.” Then run the same question on three settings.

🎬

How to run it

1) Set selector to Quick response, paste prompt A. 2) Open a fresh chat, set to Think deeper, paste the same prompt A. 3) Paste prompt B in Auto — let it route.

Prompt A — the “same question, two brains” moment

A genuinely cross-cutting business question. Quick gives a clean answer, Think Deeper structures, weighs, names risks.

“Same prompt, two models. Watch what changes — not just the words, the shape of the answer.” 
I'm trying to make the case to our board for one group-wide AI programme instead of letting each brand do their own thing. What are the strongest arguments on both sides? Don't sugarcoat the messy side.

Prompt B — let Auto decide

Tricky middle-ground task. Auto-routing should silently pick the reasoning model.

What's actually happening with EU heating policy right now? What's in force, what's coming in the next 18 months, and what should a European manufacturer be paying attention to that isn't obvious from the headlines? Cite sources.

Prompt C — the small-task case for Quick

Not every prompt needs deep reasoning. Show the audience that picking the right model is a skill.

Five subject line ideas for an internal email announcing we're kicking off a group-wide AI programme. Confident, not hyped, under 8 words each.

The dataset that lands in this room — heat pump installed capacity, 1990-2024

For a heating-systems manufacturer with three AI leaders at the table, there is exactly one public dataset that hits the bullseye: Eurostat nrg_inf_hptc — Heat pumps, installed thermal capacity by country (last update 13.05.2026). It tracks every European country’s installed MW of aerothermal, geothermal and hydrothermal capacity from 1990 to 2024. The growth curves are dramatic and unimpeachable. The bundle file: Eurostat_Heat_Pump_Installed_Capacity_1990-2024.xlsx — six sheets, EU aggregates highlighted, README explains every column.

Companion file for the cross-dataset prompt: Eurostat_Heating_Degree_Days_1979-2024.xlsx — annual HDD per country, the standard metric the heating industry uses to compare markets.

How to run it

  • In Copilot Chat, select Analyst from the Agents pane on the left.
  • Upload both Eurostat workbooks from the demo bundle — the heat-pump capacity file and the heating-degree-days file.
  • Run the three prompts in sequence — each one shows Analyst doing something Excel-Copilot can’t.
  • End by asking “show me the Python code you used” — that’s the moment for the technical person in the room.
XLSEurostat_Heat_Pump_Installed_Capacity_1990-2024.xlsx↓ XLSEurostat_Heating_Degree_Days_1979-2024.xlsx↓

1. The number a board chair can’t calculate in their head

CAGR + inflection point. Excel-Copilot doesn’t do this — Analyst writes pandas, runs it, explains the result.

“Compound growth rate per country, automatically. And it told me when the curve bent. That’s the question I’d open the board meeting with.” 
Take the "All heat pumps" sheet. For each of the top 10 European markets by 2024 installed capacity, calculate the 10-year compound annual growth rate (2014 to 2024). Then for each of those countries, find the year where growth clearly accelerated — the inflection point — and put it in the same table. Sort by CAGR. Give me one sentence per country on what likely caused the inflection.

2. The cross-dataset question only Analyst can answer

This is the killer move. Two different Eurostat files, joined on country, correlation analysis, outlier detection — in one conversation.

“Two datasets, joined on country. The intuition was wrong, and now I know which countries to look at.” 
Now use both files I uploaded. For each country, take from the heat-pump file the total installed capacity in MW in 2024, and from the heating-degree-days file the 10-year average HDD (the standard heating-industry climate metric). Normalise capacity per HDD so we compare like for like. Join on country. Show me a scatter plot (HDD on X, installed capacity in MW on Y) with a fitted regression line and the R-squared. Then list the three countries that are clearly above the line (over-adopters relative to climate) and the three that are clearly below (under-adopters). One sentence on each on what might explain it.

3. Show me your work

The audit moment. The room sees that this is pandas + matplotlib, reproducible, not a black box. This single follow-up is the most powerful prompt of the whole demo for the AI operationalizer.

“That’s the Python code it ran. Reproducible, auditable, and good enough that an analyst could pick it up tomorrow.” 
Show me the Python code you used for the correlation analysis. Add comments so a junior analyst can read it. Then suggest two ways the analysis could be sharpened if I had access to my own internal sales data alongside this public data.

Cowork is the early shape of what's coming next: an assistant that doesn't answer in 5 seconds, but works for minutes — or hours — on a real artifact. You can give it skills. Today we attach an M365 Copilot-style PowerPoint design skill, hand it a brief, and come back to a finished deck.

⏳

Open Cowork in a fresh conversation. Confirm the m365-copilot-deck skill is attached. Paste prompt 1. Tell the audience: “Let's leave this running in the background while we keep talking.”

Prompt 1 — the brief for Cowork

Short, human. The attached skill already knows the visual style — you don't have to spell it out.

“Cowork is going to build me a real deck while we keep talking. Let's see what shows up.” 
Build me a 6-slide executive deck on "Why AI changes the way we talk to our customers in 2026". Use the deck skill I attached. Audience: a business leadership team starting their AI journey. Tone: practical, optimistic, no buzzwords. Slide 1 a strong cover, slide 6 a single call-to-action. Show me thumbnails along the way so I can spot anything off.

Prompt 2 — iterate (after the reveal)

Show that Cowork doesn't just produce — it iterates. One round of edits, in plain language.

Looks good. Two changes: on slide 3 the stat is too generic — replace it with a "what changed for the customer" framing instead of a number. And on slide 5, the layout feels heavy — lighten the card grid and pull the title up. Re-render the thumbnails so I can see.

Prompt 3 — the “hand it off” moment

Demonstrates that the output is a real file, ready to share.

Save the final deck to my OneDrive as "AI-and-our-customers-2026-v1.pptx" and draft a 3-sentence Teams message I can send to my team with the link and what to look at first.

Facilitator — the agent that runs your meeting

Facilitator is GA in Microsoft Teams: it surfaces the agenda, tracks time, takes real-time collaborative notes, captures decisions and actions. Built-in — no Copilot Studio or Foundry needed. Best demoed live in a Teams meeting with a colleague.

🗣️

How to demo without a meeting

If you can't do a live Teams meeting in the room: open Workshop_Group_AI_Strategy_Outline.txt from the demo files, then run the prompts below as “what you would type” in the meeting chat. Talk the audience through the result.

TXTWorkshop_Group_AI_Strategy_Outline.txt↓

1. Set the agenda

Type in the Teams meeting chat (or describe what you'd type).

@Facilitator Add these agenda items: 1) Welcome and frame the choice — 5 min, 2) What we hear in the market — 10 min, 3) Three candidate narratives, discussion — 20 min, 4) Pick our top three 12-month commitments — 15 min, 5) Wrap, named owners, next checkpoint — 10 min.

2. Mid-meeting capture

Facilitator listens to the transcript and captures decisions as they're made.

@Facilitator Capture the decisions and open questions from the last 10 minutes. List actions with owners if anyone was assigned.

3. Closing — turn talk into tasks

Preview feature: Facilitator can create tasks in Planner directly. Mention this as “coming soon / in preview”.

@Facilitator Summarize what we decided today. List the action items with owners. Create a Word draft of today's decisions and a Planner task for each action. Mention that I want a status check in two weeks.

What it gives Vaillant

  • Meetings end with a real artifact (notes + actions), not a vague memory
  • Loop file lands in OneDrive automatically — searchable, owned by the org
  • No license cost on top — included with M365 Copilot
  • Adoption story: this is one of the easiest “everyone uses it within a week” wins

Honest limits

  • Scheduled meetings only — not channel/instant/calls
  • One meeting language at a time (mixed-language meetings only capture the chosen one)
  • External users can't see Facilitator notes
  • Tenant must have Loop enabled — admin setting

Teams agent

Interpreter — real-time meeting translation

Interpreter is GA in Microsoft Teams: real-time speech-to-speech translation in 9 languages, with the option to clone your voice so the translation sounds like you. For a multilingual European company with engineering, service, sales across DE/NL/FR/IT/ES, this changes how cross-border meetings happen.

🌐

What it does — one sentence

Each participant picks their language. They hear everyone else — live — translated into theirs. Optional: their voice is cloned (one-time consent) so the translated audio matches them.

Positioning

Copilot Studio vs Foundry — low-code or pro-code?

Both are Microsoft's official ways to build agents. They're not competing — they sit at different points on the build spectrum. This is the chart that keeps the conversation clean when Vaillant asks where to invest the next AI engineer's time.

🧭

The mental model from Microsoft's own docs

Microsoft positions Copilot Studio as the SaaS, low-code path for makers and business teams, and Foundry as the PaaS, pro-code path for AI engineers — with a third “containers + GPUs” option for fully custom IaaS work. Same Microsoft Cloud, same governance fabric. (Source: Cloud Adoption Framework, “Technology plan for AI agents”.)

Copilot Studio — the low-code workshop

  • Platform type: SaaS (managed end-to-end by Microsoft)
  • Who builds here: business analysts, makers, IT pros, citizen developers
  • Authoring: natural-language & drag-and-drop topics; visual conversation designer
  • Connectors: 1,500+ prebuilt Power Platform connectors plug-and-play
  • Models: built-in foundation models; can call Foundry-hosted models & Azure AI Search
  • Governance: Power Platform admin center, environments, DLP, ALM via Power Platform Pipelines
  • Sweet spot: department-level & line-of-business agents — IT help desk, HR FAQ, sales assistant, customer support triage
  • Time-to-first-agent: hours to days

Microsoft Foundry — the pro-code platform

  • Platform type: PaaS (you own the architecture, Microsoft runs the substrate)
  • Who builds here: AI engineers, data scientists, app developers
  • Authoring: SDKs (Python, .NET, TypeScript), Foundry playground, agent definitions in code
  • Connectors: Tool Catalog, MCP servers, OpenAPI specs, Azure Functions, Logic Apps
  • Models: any model — OpenAI, Anthropic, Llama, Mistral, custom fine-tunes from the catalog
  • Governance: Azure RBAC, private networking (standard setup), Azure Policy, full eval/monitoring loop
  • Sweet spot: production-grade, mission-critical, multi-agent orchestration — clinical, claims, complex data reasoning, regulated workloads
  • Time-to-first-agent: days to weeks (eval & integration loop)

The honest middle — you'll use both

The hybrid pattern Microsoft documents

Copilot Studio's low-code interface calling Foundry's advanced models. Business teams ship faster than full pro-code; they still get enterprise-grade reasoning. That's the documented “reduces development time vs full pro-code, more customization than pure SaaS” sweet spot.

What this means for Vaillant

  • Don't pick a camp. Run both, with a clear handoff rule.
  • 1st-mover use cases (HR, IT, support FAQs) → Copilot Studio
  • Use cases that touch product data, customer cases, regulatory text → Foundry
  • Both publish into Copilot Chat — the business user never sees the seam
🔗

The bridge sentence for the room

“Copilot Studio is where your makers build agents in days. Foundry is where your engineers build agents that run the business. Both land in the same Copilot UI — that's not two strategies, that's one.”

The dev process matters — especially for live, customer-facing agents

If the agent runs in customer support, you need three versions of it living side-by-side: Production (always on, never touched directly), Test/UAT (the next release, business validates here), Development (where engineers and makers actually change things). Both platforms support that — the how is the part that defines what kind of team you need.

Copilot Studio — ALM on Power Platform

  • Environments: minimum 3 — Dev & Test as sandbox environments, Production as a production-type environment, secured via Entra security groups
  • Containers: solutions carry the agent + dependencies between environments; environment variables & connection references swap settings/secrets per stage
  • Source control: native Git integration (Azure DevOps repos, GA Nov 2024) — commits in YAML format, no command line needed
  • Local dev: Copilot Studio VS Code extension — clone agent, edit locally, push back, optional “reattach” to a different environment
  • CI/CD options: Power Platform Pipelines (low-code, hours to set up) · GitHub Actions for Power Platform · Azure DevOps with Power Platform Build Tools · Copilot ALM Starter (prebuilt GitHub Actions, PR validation, workload identity federation)
  • Golden rules (from Microsoft docs): never customize in Test or Production, always work inside a solution, deploy as managed, use environment variables for secrets
  • Who runs this loop: makers + a Power Platform admin + (for the advanced pattern) a DevOps engineer for the initial pipeline

Foundry — ALM as code on Azure

  • Environments: separate Foundry projects (or resource groups / subscriptions) per stage — full isolation, full Azure RBAC, private networking per stage
  • Agent definition lives as code: agent.yaml, Python/.NET/TypeScript SDK, prompts & tool configs in the repo — everything diffs in a normal pull request
  • Source control: Git (GitHub or Azure DevOps) as the source of truth from day one — branches per feature, PR reviews, branch protection
  • CI/CD: standard GitHub Actions or Azure Pipelines · Bicep / azd up for infra · container build → ACR push → hosted agent deploy · eval gates on PR (regression testing against datasets) · canary / blue-green rollout patterns
  • Observability: Application Insights + Foundry tracing wired into the SDK — same telemetry stack engineers already use
  • Who runs this loop: AI engineers / app developers / SRE — the team that already does CI/CD for production services

Sources: Copilot Studio — Establish an ALM strategy · Native Git integration for Power Platform · Clone your Copilot Studio agent in VS Code · Enterprise ALM reference architecture.

Rate limits & scale — where the platform pushes back

For a customer-service agent that handles real volume, throttling is not a footnote — it’s a hard ceiling. Both platforms have limits; they sit in different places.

Dimension Copilot Studio Foundry
Message ceiling 8,000 requests/min to a single agent per Dataverse environment No fixed agent-level ceiling — scales with model deployment quota
Generative AI calls 50–100 RPM & 1,000–2,000 RPH per environment, depending on prepaid message packs (more packs = higher ceiling) Tokens-per-minute (TPM) & requests-per-minute (RPM) set per model deployment; you choose the size
How you raise it Buy more capacity packs, switch to pay-as-you-go, or open a support ticket (PAYG only — not guaranteed) Increase the deployment quota, deploy across more regions, or buy Provisioned Throughput Units (PTU) for guaranteed capacity
Scope of the limit Per Dataverse environment — one chatty agent can starve others sharing the env Per model deployment — isolation by design
Billing model Copilot Credits (per message/action) or capacity packs Per token (PAYG) or per PTU per hour (reserved)
What happens at the limit End user sees “agent currently unavailable” or “usage limit reached” HTTP 429 to the calling code — your app decides how to handle it (retry, queue, fallback model)

Sources: Copilot Studio quotas and limits · Resolve throttling errors in agents · Azure AI Foundry quotas and limits · Provisioned Throughput Units (PTU).

🚦

The honest implication for customer-facing workloads

If the agent answers questions inside the company at normal office volume — Copilot Studio absorbs that comfortably. If the agent is on the front line of customer service at peak hours — campaign weekend, product launch, a regulatory headline — you eventually hit Copilot Studio’s per-environment ceiling and have nowhere to go but support tickets and more packs. That’s the moment the pro-code path on Foundry pays for itself: deployment-level quotas you control, PTU for guaranteed throughput, multi-region failover, and an HTTP 429 your code can actually react to instead of a generic error in the user’s chat window.

Vaillant GPT is the engine in one room. Copilot is the building that connects every room — Outlook, Word, Excel, Teams — with audit, identity, file context. The GPT becomes one of the agents people call, not the only door.

Enterprise Data Protection on by default. Purview audit logs every interaction. DLP labels block sensitive content from leaving. Conditional Access via Entra. Tell them: “Many German enterprises have already cleared this with their works councils — we'll bring the deck.”

“Models change every month — aren't we chasing a moving target?”

Yes — and that's why model choice is part of the platform, not part of your code. You pick the model per task. Microsoft handles version updates, deprecations, evaluations. You focus on use cases.

“What's the smallest first step after the summer?”

A 6-week pilot: 25-50 licensed business users, 3 named use cases, weekly success metrics. Pair it with one Copilot Studio agent that uses Vaillant data. End with a steering-committee report that decides scale-out.
